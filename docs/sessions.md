# ADK Session Management & Data Architecture

This document describes how sessions, states, and event logs are structured, stored, and managed in BotsonV2 and the underlying Agent Development Kit (ADK) framework.

---

## 1. Data Schema & Scoping

All sessions are persisted in a GORM-managed SQLite database (`sessions.db` in `~/.botsonv2/`). The schema utilizes a composite primary key layout to isolate session histories.

### The Composite Key
Every session is uniquely identified by three fields:
1. `app_name` (string): The identifier of the agent running the session (e.g., `general_assistant`).
2. `user_id` (string): The identifier of the user workspace. 
   - *Note:* The ADK built-in console hardcodes this to `user`. Our Custom Chat and Workspace Dashboard are aligned to use `user` to keep lists and stats consistent.
3. `id` (string): A standard UUID generated by the backend when the session is initialized.

Because GORM queries search on all three keys, querying for user `user` will not return sessions created under other user IDs (like the old `default` test ID).

---

## 2. Session State & Metadata

A session's state is stored in the `sessions` table under a GORM Datatype JSON map.

```json
{
  "__session_metadata__": {
    "displayName": "What tools do you have?"
  }
}
```

### Display Names & Titles
- **Automatic Renaming:** When starting a new chat session, the Custom Chat UI performs a lazy session initialization on the first user message. It calls the session creation endpoint passing the user's message as `__session_metadata__.displayName`.
- **List Rendering:** Both the Custom Chat UI and the Workspace Dashboard retrieve the display name from the JSON state. If present, it replaces the raw UUID in list views to provide a clean, human-readable title.

---

## 3. Events & History

A session has a Has-Many relationship with `events` (stored in the `events` table). Events are linked via the same composite keys (`app_name`, `user_id`, `session_id`).

Each event records:
- **`ID`**: Unique event ID.
- **`Author`**: Who generated the event (e.g., `user`, or the active agent like `general_assistant`).
- **`Timestamp`**: High-precision timestamp (truncated to microsecond precision in SQLite to prevent rounding discrepancies).
- **`Content`**: The actual message parts (text, image data, audio PCM streams).
- **`Output`**: GORM JSON output for tool executions and tracing.

---

## 4. REST API Endpoint Reference

The unified backend on port `8080` exposes the following endpoints for session management:

### Create Session
- **Endpoint:** `POST /api/apps/{app_name}/users/{user_id}/sessions`
- **Body:** `{ "state": {}, "events": [] }`
- **Behavior:** Creates a new session record on the backend. Generates a standard UUID and returns the session object.

### Get Session Details
- **Endpoint:** `GET /api/apps/{app_name}/users/{user_id}/sessions/{session_id}`
- **Behavior:** Returns the session state and all chronological history events.

### List Sessions
- **Endpoint:** `GET /api/apps/{app_name}/users/{user_id}/sessions`
- **Behavior:** Returns an array of all sessions (including states and update times) registered under the given agent and user ID.

### Delete Session
- **Endpoint:** `DELETE /api/apps/{app_name}/users/{user_id}/sessions/{session_id}`
- **Behavior:** Atomically deletes the session record and cascades deletion to all associated event records in GORM.
